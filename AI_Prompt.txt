You are to develop a single-guild Discord bot using Hikari (Python 3.13) with lightbulb (for commands) and miru (for components/views). The bot provides a reaction role system. Write production-ready, well-commented code with proper error handling and performance optimisations.

General Constraints
Slash commands only – no prefix commands (!, ?, etc.).
Single guild – all commands and configurations are tied to one guild ID (provided in a config file).
Optimised for speed – use ephemeral responses, defer when needed, avoid blocking I/O.
Persistent – configurations survive restarts using JSON storage.
Support both button-based and emoji-based role assignment – the admin chooses per embed.
Framework Requirements (Hikari Stack)
Core: hikari
Commands: lightbulb
Components (buttons/selects): miru
Configuration File (server_config.json)

Located in the project root (next to main.py).

If the file is missing:

Create it with default/empty values.
Prompt the user via console to fill it.
Do not crash.

On startup:

Validate bot token and guild ID.
If missing → log error and exit gracefully.
Data Storage (data/reaction_roles.json)
Store each reaction role message configuration.
Auto-create missing folders/files.
If corrupted:
Backup as reaction_roles.json.corrupt.<timestamp>
Reset safely.
Slash Commands (Admin Only)

Use lightbulb slash commands.

Commands:

/create_reaction_role
/edit_reaction_role

Permission check:

User must have at least one role from admin_role_ids.
Otherwise → ephemeral error response.

Delete user message

Role select (required)
Style select (buttons only)
Confirm button

Defer interaction
Save mapping
Update wizard message

Build embed

Add reactions manually

Logging & Error Handling

Use Python logging:

Bot ready

No spam logs.

Catch all critical errors.

Performance Optimisations
Cache JSON in memory
Save with asyncio.to_thread
Always defer long interactions
Use cached members (guild.get_member)
Avoid REST calls unless needed
Critical Implementation Notes (Hikari Version)
Modals = text inputs only
Components must use miru Views

Always acknowledge interactions:

await interaction.create_initial_response(...)

Use followups:

await interaction.execute(...)
Delete temporary messages after use
Always update main wizard message, not temporary ones
